# Стандарт: Политика No-Unicode v2.3

> ID: STD-DOC-003
> Version: 2.3.0
> Level: **[C] Critical** + **[W] Warning** + **[I] Info**
> Related: Markdown Standard (STD-DOC-002)

Стандарт использования символов, иконок и графики. Уровень Design System / Engineering Governance.

---

## 1. Назначение

Данный стандарт устанавливает правила использования графических символов Unicode во всех слоях продукта: интерфейс, контент, код, системные коммуникации.

### Цели:

- обеспечение визуальной согласованности
- поддержание профессионального уровня продукта
- гарантия контроля через дизайн-систему
- исключение неконтролируемых визуальных артефактов
- обеспечение автоматического принудительного контроля через ESLint там, где это возможно

---

| **MARKDOWN_STANDARD v2.3** | [C] Critical | README.md, документация проекта |

## 3. Уровни строгости

Стандарт применяет **поэтапный подход** вместо абсолютного запрета:

| Уровень | Обозначение | Контекст | Действие |
|---------|-------------|----------|----------|
| Critical | [C] | UI, production код | Блокирует merge |
| Warning | [W] | Только AI-chat | Предупреждение в review (см. MARKDOWN_STANDARD) |
| Info | [I] | Внутренние заметки, прототипы | Рекомендация |

### Применение уровней:

| Контекст | Уровень | Обоснование |
|----------|---------|-------------|
| UI-компоненты | [C] Critical | Прямое влияние на пользователя |
| API-ответы, ошибки | [C] Critical | Отображаются в интерфейсе |
| Production код | [C] Critical | Может попасть в UI |
| Документация проекта | [C] Critical | См. MARKDOWN_STANDARD |
| README-файлы | [C] Critical | См. MARKDOWN_STANDARD |
| Внутренние заметки | [I] Info | Только для разработчиков |
| Прототипы / MVP | [I] Info | Временный код |
| Логи, отладка | [I] Info | Не видны пользователю |
| AI-коммуникация (чат) | [W] Warning | Профессиональный стиль общения агента |

**Примечание для AI-коммуникации:** Ответы AI-агента в чате не должны содержать эмодзи и графику Unicode. Это обеспечивает единообразный профессиональный стиль наряду с кодом и документацией. Пользовательские сообщения не регулируются данным стандартом.

## 5. Причины ограничений

| Проблема | Описание |
|----------|----------|
| Несогласованный рендер | Unicode отображается по-разному на разных ОС, браузерах, устройствах |
| Отсутствие контроля | Невозможно централизованно изменять стиль, управлять темами |
| Отсутствие масштабируемости | Нет контроля размера, нет адаптивности |
| Нарушение профессионального стандарта | Снижает доверие, нарушает визуальную иерархию |
| Сложность автоматического принудительного контроля | Символы Unicode в коде невозможно надёжно выявить только ручным просмотром |

## 8. AI-промпты

### 8.1. Корректная формулировка

```text
Output must contain only:
- ASCII characters (a-z, A-Z, 0-9, standard punctuation)
- Cyrillic characters (a-ya, A-YA)
- Whitelisted diagram symbols (for [I] level): -> <- => <= \| + - v ^ > <
```

### 8.2. Очистка документа перед анализом

```javascript
// Remove emoji and Unicode graphics
text.replace(/[\u{1F000}-\u{1FFFF}]|[\u{2600}-\u{27BF}]|[\u{FE00}-\u{FEFF}]|[\u{1F900}-\u{1F9FF}]|[\u{2702}-\u{27B0}]/gu, '')
```

### 8.3. Финальная санитизация

```javascript
// For [C] level (code/UI) - ASCII + Cyrillic only (typography strictly prohibited)
text.replace(/[^\x20-\x7E\u0400-\u04FF]/g, '')

// For [I] level - with whitelist for diagrams
// Note: 'v' removed from explicit enumeration as it's in basic ASCII range \x20-\x7E
text.replace(/[^\x20-\x7E\u0400-\u04FF\-\>\<\=\|\+\^]/g, '')

// ATTENTION: Level [W] (documentation) is regulated by MARKDOWN_STANDARD v2.3.
// There typographic characters (em dash, en dash, degree, copyright) ARE ALLOWED in plain text, therefore this
// strict sanitization is NOT applied to .md files.
```

### 8.4. Требования к диалогу в чате

Данный подраздел определяет правила ведения диалога AI-агента с пользователем (контекст `[W] Warning` из раздела 3). Правила распространяются на каждое сообщение, которое агент отправляет в канал чата: обычные ответы, блоки кода, таблицы, списки, структурированные отчёты, сводки журнала работ и описания вызовов инструментов. Пользовательские сообщения НЕ регулируются — только вывод агента.

**8.4.1. Набор символов для вывода в чат**

| Разрешено | Запрещено |
|-----------|-----------|
| Латинские буквы, цифры, стандартная пунктуация | Emoji (любой диапазон, включая ZWJ-последовательности) |
| Кириллические буквы (русский, украинский и др.) | Unicode-пиктограммы (стрелки, звёзды, галочки, крестики) |
| ASCII-символы для диаграмм из белого списка только внутри блоков кода: `<` `>` `=` `\|` `+` `-` `^` `v` | Dingbats, Miscellaneous Symbols, Mathematical Operators, используемые как декорация |
| Типографские символы из базового набора STD-DOC-003 §6.1 (тире, короткое тире, градус, плюс-минус) ТОЛЬКО в обычном тексте | Типографские символы в блоках кода, заголовках, именах файлов, заголовках таблиц |
| Текстовые теги статусов: `[OK]`, `[FAIL]`, `[TODO]`, `[WARNING]`, `[INFO]`, `[DRAFT]`, `[BLOCKED]`, `[DEPRECATED]` | Unicode-иконки статусов: галочка, крестик, песочные часы, замок, знак предупреждения |

**8.4.2. Структурные правила для сообщений в чате**

1. **Ни эмодзи как маркеры списков или индикаторы статуса.** Используйте `-` для неупорядоченных списков (согласно STD-DOC-002 §5.2) и текстовые теги из §8.4.1 для статусов.
2. **Ни Unicode-псевдографика для деревьев или рамок в тексте чата.** Структуры деревьев (файловые деревья, иерархии вызовов) ДОЛЖНЫ использовать ASCII-отступы с `+--`, `|` и `\-` внутри экранированного блока кода `text`. Символы рисования рамок (U+2500–U+257F) ЗАПРЕЩЕНЫ в выводе чата.
3. **Блоки кода ДОЛЖНЫ указывать язык** (согласно STD-DOC-002 §5.4). Неизвестный язык — используйте `text` или `bash`. Это применяется одинаково к встроенным примерам команд и к многострочному коду.
4. **Пути к файлам в чате ДОЛЖНЫ быть заключены** в обратные кавычки (например, `src/lib/utils.ts`) — никогда не оборачивайте в угловые скобки и не декорируйте стрелками.
5. **Заголовки в длинных ответах чата** (более ~30 строк) ДОЛЖНЫ использовать синтаксис Markdown `##` / `###`, а не Unicode-декораторы или горизонтальные линии из `━`, `═` и т.д.
6. **Таблицы ДОЛЖНЫ использовать стандартный синтаксис pipe-таблиц Markdown** — никогда не используйте символы рисования рамок Unicode для имитации границ таблиц.

**8.4.3. Тон и темп**

- Агент отвечает на языке ввода пользователя (согласно правилу языковой согласованности проекта).
- Код, пути к файлам, идентификаторы, имена команд и ключи конфигурации остаются на языке оригинала (английский, латынь и т.д.), даже если окружающий текст чата на другом языке.
- Когда агент допускает ошибку или обнаруживает несоответствие в своём предыдущем выводе, он ДОЛЖЕН явно указывать на исправление обычным текстом ("Correction:", "Update:"), а не с помощью эмодзи или символов.
- Когда агент отказывается от запроса (безопасность, политика, возможности), отказ ДОЛЖЕН быть обычным текстом с кратким обоснованием — без знаков предупреждения, без зачёркнутых эмодзи.

**8.4.4. Запрещённые паттерны чата**

| Паттерн | Почему запрещён | Правильная замена |
|---------|-----------------|-------------------|
| `✅ Done` или `❌ Failed` как строка статуса | Unicode-пиктограмма в чате | `[OK] Done` / `[FAIL] Failed` |
| `→` (стрелка) как маркер или разделитель | Unicode-стрелка как декорация | `->` (внутри блока кода) или `:` (в обычном тексте) |
| `📁 Folder:` или `🔒 Locked:` | Эмодзи как маркер раздела | `Folder:` / `Locked:` или `[LOCKED]` |
| Рамочное дерево: `├── src` `│   └── ...` | Unicode-псевдографика в чате | ASCII-дерево в блоке `text`: `+-- src` `\  +-- ...` |
| `⚠️ Warning: ...` | Эмодзи как маркер предупреждения | `[WARNING] ...` |
| Горизонтальная линия из `━` × N | Unicode-декорация | Markdown `---` |
| Таблица с `│` и `─` границами | Unicode-псевдографическая таблица | Стандартная pipe-таблица Markdown |

**8.4.5. Принудительное соблюдение**

Диалог в чате проверяется на уровне `[W] Warning`: нарушения не блокируют ответ агента, но проверяются в периодических аудитах качества и отмечаются в записи журнала работ агента за сессию. Повторяющиеся нарушения эскалируются к Tech Lead в соответствии с разделом 9.1 STD-DOC-002.

**8.4.6. Связь с другими разделами**

- §4 (Что запрещено) определяет категории символов — §8.4 сужает их до вывода в чат.
- §6 (Разрешённые символы) определяет белый список — §8.4.1 переопределяет релевантную для чата подмножество.
- §10.1 (Правило ESLint `no-unicode-policy`) НЕ запускается на выводе чата (чат — это не файл). Принудительное соблюдение осуществляется вручную / на основе аудита.
- §13.3 (Чеклист соответствия для AI-коммуникации) является операционным чеклистом — §8.4 является нормативной спецификацией, стоящей за ним.

## 9. Стратегия резервного копирования

### 9.1. Когда SVG недоступен

| Ситуация | Решение |
|----------|---------|
| Ошибка загрузки SVG | Текстовая альтернатива (скрытая, aria-label) |
| Почтовые клиенты | Текст + стилизованный span |
| Терминал / CLI | Текст + ANSI-цвета |
| Обычный текст | Только текст |

### 9.2. Реализация резервного копирования

```html
<!-- SVG with fallback via onerror -->
<span class="icon">
  <svg onerror="this.style.display='none';this.nextElementSibling.style.display='inline'"
       aria-hidden="true">
    <!-- icon content -->
  </svg>
  <span class="icon-fallback" style="display:none">Save</span>
</span>
```

```css
.icon-fallback {
  display: none;
}
/* Fallback shown via JS onerror on SVG element */
```

## 10. Контроль и принудительное соблюдение

### 10.1. Пользовательское правило ESLint: `no-unicode-policy`

Это основной механизм автоматического принудительного соблюдения политики No-Unicode. Файл правила находится в `eslint-rules/no-unicode-policy.js` в корне проекта и экспортирует несколько подправил, охватывающих различные контексты и уровни строгости.

#### 10.1.1. Полная реализация правила

```javascript
// eslint-rules/no-unicode-policy.js
module.exports = {
  configs: {
    recommended: {
      plugins: ["no-unicode-policy"],
      rules: {
        "no-unicode-policy/no-emoji": "error",
        "no-unicode-policy/no-unicode-graphics": "error",
        "no-unicode-policy/no-emoji-in-md": "warn",
        "no-unicode-policy/no-unicode-graphics-in-md": "warn",
      },
    },
  },
  rules: {
    // -------------------------------------------------------
    // Rule 1: no-emoji
    // Context: Production code (.ts, .tsx, .js, .jsx)
    // Level: [C] Critical — blocks merge
    // Detects emoji in string literals, template literals, and JSX text nodes.
    // -------------------------------------------------------
    "no-emoji": {
      meta: {
        type: "problem",
        docs: {
          description:
            "Disallow emoji in production code (STD-DOC-003 section 4.1, level [C])",
          category: "Unicode Policy",
          recommended: true,
        },
        messages: {
          emojiInLiteral:
            "Emoji are prohibited in production code [C]. Use SVG icons or text alternatives instead. (STD-DOC-003 section 4.1)",
          emojiInTemplate:
            "Emoji are prohibited in template literals [C]. Use SVG icons or text alternatives instead. (STD-DOC-003 section 4.1)",
          emojiInJSX:
            "Emoji are prohibited in JSX text [C]. Use SVG icons or text alternatives instead. (STD-DOC-003 section 4.1)",
        },
      },
      create(context) {
        const emojiPattern =
          /[\u{1F600}-\u{1F64F}\u{1F300}-\u{1F5FF}\u{1F680}-\u{1F6FF}\u{1F1E0}-\u{1F1FF}\u{2600}-\u{27BF}\u{FE00}-\u{FEFF}\u{1F900}-\u{1F9FF}\u{1FA00}-\u{1FA6F}\u{1FA70}-\u{1FAFF}\u{2702}-\u{27B0}]/u;

        return {
          Literal(node) {
            if (typeof node.value === "string" && emojiPattern.test(node.value)) {
              context.report({ node, messageId: "emojiInLiteral" });
            }
          },
          TemplateLiteral(node) {
            for (const quasi of node.quasis) {
              if (
                quasi.value &&
                quasi.value.cooked &&
                emojiPattern.test(quasi.value.cooked)
              ) {
                context.report({ node, messageId: "emojiInTemplate" });
                break;
              }
            }
          },
          JSXText(node) {
            if (node.value && emojiPattern.test(node.value)) {
              context.report({ node, messageId: "emojiInJSX" });
            }
          },
        };
      },
    },

    // -------------------------------------------------------
    // Rule 2: no-unicode-graphics
    // Context: Production code (.ts, .tsx, .js, .jsx)
    // Level: [C] Critical — blocks merge
    // Detects Unicode graphic characters (beyond emoji) used as visual elements:
    // box-drawing characters, block elements, geometric shapes, arrows, etc.
    // -------------------------------------------------------
    "no-unicode-graphics": {
      meta: {
        type: "problem",
        docs: {
          description:
            "Disallow Unicode graphic characters in production code (STD-DOC-003 section 4.1, level [C])",
          category: "Unicode Policy",
          recommended: true,
        },
        messages: {
          unicodeGraphicsInLiteral:
            "Unicode graphic characters are prohibited in production code [C]. Use SVG icons instead. (STD-DOC-003 section 4.1)",
          unicodeGraphicsInTemplate:
            "Unicode graphic characters are prohibited in template literals [C]. Use SVG icons instead. (STD-DOC-003 section 4.1)",
          unicodeGraphicsInJSX:
            "Unicode graphic characters are prohibited in JSX text [C]. Use SVG icons instead. (STD-DOC-003 section 4.1)",
        },
      },
      create(context) {
        // Box-drawing, block elements, geometric shapes, arrows, misc symbols
        const unicodeGraphicsPattern =
          /[\u{2500}-\u{257F}\u{2580}-\u{259F}\u{25A0}-\u{25FF}\u{2190}-\u{21FF}\u{2200}-\u{22FF}\u{2300}-\u{23FF}\u{2400}-\u{243F}\u{2440}-\u{244A}\u{25CC}\u{2800}-\u{28FF}\u{2B50}\u{2B55}\u{2934}\u{2935}]/u;

        return {
          Literal(node) {
            if (
              typeof node.value === "string" &&
              unicodeGraphicsPattern.test(node.value)
            ) {
              context.report({ node, messageId: "unicodeGraphicsInLiteral" });
            }
          },
          TemplateLiteral(node) {
            for (const quasi of node.quasis) {
              if (
                quasi.value &&
                quasi.value.cooked &&
                unicodeGraphicsPattern.test(quasi.value.cooked)
              ) {
                context.report({ node, messageId: "unicodeGraphicsInTemplate" });
                break;
              }
            }
          },
          JSXText(node) {
            if (node.value && unicodeGraphicsPattern.test(node.value)) {
              context.report({ node, messageId: "unicodeGraphicsInJSX" });
            }
          },
        };
      },
    },

    // -------------------------------------------------------
    // Rule 3: no-emoji-in-md
    // Context: Markdown files (.md)
    // Level: [W] Warning — does not block merge
    // Scans raw .md text (excluding code blocks) for emoji.
    // -------------------------------------------------------
    "no-emoji-in-md": {
      meta: {
        type: "suggestion",
        docs: {
          description:
            "Disallow emoji in Markdown documentation (STD-DOC-003 section 4.1 + STD-DOC-002, level [C])",
          category: "Unicode Policy",
          recommended: true,
        },
        messages: {
          emojiInMd:
            "Emoji are prohibited in documentation [W]. Use text tags like [OK], [FAIL] instead. (STD-DOC-003 section 4.1, STD-DOC-002 section 4.4)",
        },
      },
      create(context) {
        const emojiPattern =
          /[\u{1F600}-\u{1F64F}\u{1F300}-\u{1F5FF}\u{1F680}-\u{1F6FF}\u{1F1E0}-\u{1F1FF}\u{2600}-\u{27BF}\u{FE00}-\u{FEFF}\u{1F900}-\u{1F9FF}\u{1FA00}-\u{1FA6F}\u{1FA70}-\u{1FAFF}\u{2702}-\u{27B0}]/u;

        return {
          Program(node) {
            const sourceCode =
              context.sourceCode || context.getSourceCode();
            const text = sourceCode.getText();

            // Remove fenced code blocks — they are handled by the
            // code-block-level rules (no-emoji, no-unicode-graphics)
            const textWithoutCodeBlocks = text.replace(
              /```[\s\S]*?```/g,
              ""
            );

            const lines = textWithoutCodeBlocks.split("\n");
            lines.forEach((line, index) => {
              if (emojiPattern.test(line)) {
                context.report({
                  loc: { line: index + 1, column: 0 },
                  messageId: "emojiInMd",
                });
              }
            });
          },
        };
      },
    },

    // -------------------------------------------------------
    // Rule 4: no-unicode-graphics-in-md
    // Context: Markdown files (.md)
    // Level: [W] Warning — does not block merge
    // Scans raw .md text (excluding code blocks) for Unicode graphics.
    // -------------------------------------------------------
    "no-unicode-graphics-in-md": {
      meta: {
        type: "suggestion",
        docs: {
          description:
            "Disallow Unicode graphic characters in Markdown documentation (STD-DOC-003 section 4.1 + STD-DOC-002, level [W])",
          category: "Unicode Policy",
          recommended: true,
        },
        messages: {
          unicodeGraphicsInMd:
            "Unicode graphic characters are prohibited in documentation [W]. Use ASCII art in code blocks or text alternatives. (STD-DOC-003 section 4.1, STD-DOC-002 section 6.2)",
        },
      },
      create(context) {
        const unicodeGraphicsPattern =
          /[\u{2500}-\u{257F}\u{2580}-\u{259F}\u{25A0}-\u{25FF}\u{2190}-\u{21FF}\u{2200}-\u{22FF}\u{2300}-\u{23FF}]/u;

        return {
          Program(node) {
            const sourceCode =
              context.sourceCode || context.getSourceCode();

            // Remove fenced code blocks
            const textWithoutCodeBlocks = text.replace(
              /```[\s\S]*?```/g,
              ""
            );

            const lines = textWithoutCodeBlocks.split("\n");

              if (unicodeGraphicsPattern.test(line)) {
                context.report({
                  loc: { line: index + 1, column: 0 },
                  messageId: "unicodeGraphicsInMd",
                });
              }
            });
          },
        };
      },
    },
  },
```

#### 10.1.2. Архитектура правила и маппинг уровней строгости

Правила ESLint в `no-unicode-policy.js` разработаны для отражения поэтапных уровней строгости из раздела 3 данного стандарта. Маппинг между уровнями стандарта и серьёзностью ESLint является фундаментальным для понимания того, как работают вложенные стандарты:

| Уровень стандарта | Серьёзность ESLint | Влияние на CI | Пример правила |
|-------------------|-------------------|---------------|----------------|
| [C] Critical | `error` | Блокирует merge | `no-unicode-policy/no-emoji` |
| [W] Warning | `warn` | Сообщает, но не блокирует | `no-unicode-policy/no-emoji-in-md` |
| [I] Info | `off` | Не проверяется ESLint | Нет правила (только ручная договорённость) |

**Почему этот маппинг важен:** Одно и то же нарушение (например, символ эмодзи) вызывает разную серьёзность ESLint в зависимости от контекста. Эмодзи в React-компоненте — это `error` (он попадёт в UI — [C] Critical). Тот же эмодзи в README — это `warn` (это документация — [W] Warning). Реализация правила учитывает контекст через отдельные имена правил, а не через логику серьёзности времени выполнения.

### 10.2. Интеграция конфигурации ESLint

Правило `no-unicode-policy.js` разработано для работы как с плоской конфигурацией (ESLint 9+), так и с_legacy-конфигурацией (ESLint 8). Полные примеры конфигурации см. в **STD-DOC-002 раздел 10.3**. Ключевые моменты интеграции:

1. **Файлы production-кода** (`.ts`, `.tsx`, `.js`, `.jsx`): Правила `no-emoji` и `no-unicode-graphics` установлены в `error`
2. **Markdown-файлы** (`.md`): Правила `no-emoji-in-md` и `no-unicode-graphics-in-md` установлены в `warn`
3. **Блоки кода внутри Markdown** (`.md/**`): Применяются production-правила — `no-emoji` и `no-unicode-graphics` установлены в `error`, поскольку примеры кода в документации должны соответствовать тем же стандартам, что и production-код

Эта трёхуровневая конфигурация обеспечивает корректное применение вложенных стандартов на каждом уровне без конфликтов между ними.

### 10.3. Взаимодействие с `no-irregular-whitespace`

Встроенные правила ESLint `no-irregular-whitespace` обнаруживает некоторые символы пробелов Unicode (NBSP, ZWSP, разделители строк/абзацев и т.д.). Это правило дополняет пользовательские правила `no-unicode-policy`:

| Правило | Что обнаруживает | Где применяется |
|---------|------------------|-----------------|
| `no-unicode-policy/no-emoji` | Пиктограммы emoji | Строковые литералы, шаблонные литералы, JSX |
| `no-unicode-policy/no-unicode-graphics` | Рисование рамок, стрелки, геометрические фигуры | Строковые литералы, шаблонные литералы, JSX |
| `no-irregular-whitespace` | NBSP, ZWSP, разделители строк/абзацев, BOM | Весь исходный код |

**Рекомендация:** Включите `no-irregular-whitespace: "error"` наряду с пользовательскими правилами. Оно обнаруживает проблемы с пробелами Unicode, которые не покрываются пользовательскими правилами.

4. При необходимости добавьте в белый список

### 11.4. Политика инлайн-отключения ESLint

Когда исключение согласовано (раздел 11.3), соответствующее правило ESLint можно отключить инлайн. Комментарий отключения должен ссылаться на согласованное исключение.

```typescript
// eslint-disable-next-line no-unicode-policy/no-emoji -- Approved exception: ISSUE-1234, email campaign requires emoji
const emailSubject = "Welcome to our platform!";
```

**Правила инлайн-отключений:**

1. Каждое `eslint-disable` должно ссылаться на конкретный номер согласованного issue
2. Отключения правил уровня [C] без согласованного issue отклоняются при просмотре кода
3. `eslint-disable` без указания конкретного имени правила запрещено
4. Отключения должны охватывать как можно меньший диапазон (предпочтительно `eslint-disable-next-line` вместо `eslint-disable`)

## 14. Формат Stack Signature

- Размещение: правый нижний угол (только для корневых `README.md` и `CHANGELOG.md`)
- Формат: `Built with: <технологии проекта>` (конкретный стек определяется проектом, а не стандартом)
- Пример (для проектов на Next.js): `Built with: Next.js 16 + TypeScript + Tailwind CSS`
- Разрешено: латиница, кириллица, цифры, символы `+` и `:`
- Запрещено: эмодзи, Unicode-графика

**Примечание:** Стандарт определяет формат (структуру), а не конкретные технологии. Значение по умолчанию для проектов этого стека см. в `templates/README_TEMPLATE.md`.

## 15A. Известные проблемы и предложенные решения

Данный документ описывает обнаруженные несоответствия, отсутствующий контент и предложенные коррективы. Каждая проблема имеет ID, статус и предложенное действие. Проблемы, решённые в текущей версии, отмечены `[RESOLVED]`; нерешённые проблемы отмечены `[OPEN]`.

### UNI-001 `[RESOLVED in v2.3.0]` — Отсутствует нормативный раздел для диалога в чате

**Проблема:** До v2.3.0 диалог в чате упоминался только в §3 (Уровни строгости) как одна строка контекста `[W]` и в §13 как чеклист из 2 пунктов. Не было нормативной специкации того, что разрешено/запрещено в выводе AI-агента в чате. У агентов не было конкретной справки для применения.

**Решение:** Добавлен §8.4 "Требования к диалогу в чате" с шестью подразделами: набор символов, структурные правила, тон/темп, таблица запрещённых паттернов, принудительное соблюдение и связь с другими разделами. Расширен чеклист §13 с 2 до 9 пунктов в соответствии с §8.4.

### UNI-002 `[OPEN]` — Правило ESLint не охватывает вывод в чате

**Проблема:** Пользовательское правило ESLint `no-unicode-policy.js` (§10.1) запускается на исходных файлах (`.ts`, `.tsx`) и Markdown-файлах (`.md`). Вывод в чате — это не файл, он никогда не проходит через ESLint. Таким образом, принудительное соблюдение осуществляется вручную и на основе аудита, что не согласуется с моделью автоматического принудительного контроля, используемой для кода и документации.

**Предлагаемое решение:** Реализовать линтер вывода чата как отдельный инструмент (например, `lint-chat.js`), который принимает ответы агента из журнала чата и выполняет те же проверки Unicode. Интегрировать в пост-ответный хук агента. До этого §8.4.5 документирует политику ручного принудительного соблюдения.

### UNI-003 `[OPEN]` — Устаревшая ссылка на версию в `DESIGN-001-design-system.md`

**Проблема:** Заголовок и §8 `DESIGN-001-design-system.md` (STD-DESIGN-001) ссылаются на "No-Unicode Policy v2.1". Фактическая версия этого стандарта — v2.3.0.

**Предлагаемое решение:** Обновить заголовок и §8 `DESIGN-001-design-system.md` для ссылки на "No-Unicode Policy v2.3". Перекрёстную ссылку см. в записи DES-001 известных проблем STD-DESIGN-001.

### UNI-004 `[OPEN]` — Несоответствие версии в реестре `META-001-standard-id-system.md`

**Проблема:** Реестр в `META-001-standard-id-system.md` (STD-META-001 §4) указывает STD-DOC-003 как v2.1.3. Фактическая версия этого стандарта — v2.3.0.

**Предлагаемое решение:** Обновить STD-META-001 §4.4 для отражения v2.3.0 для STD-DOC-003. Перекрёстную ссылку см. в записи META-002 известных проблем STD-META-001.
