## 0. TL;DR -- Краткая Справка

**Проверьте файл за 30 секунд (рекомендуется):**

```bash
bash scripts/check-md.sh path/to/file.md
```

Обертка запускает три уровня -- статические проверки (только bash), ESLint (если установлен) и `lint-md.js` (если присутствует) -- и выходит с ненулевым кодом при любом нарушении. Полная таблица этапов в §10.7.

**Или запустите базовые инструменты напрямую (тот же эффект):**

```bash
npx eslint path/to/file.md --plugin markdown --max-warnings=0
node lint-md.js --level=C path/to/file.md
```

Оба должны выйти с кодом 0. Любая ошибка блокирует коммит. Скрипт-обертка существует, чтобы участникам не нужно было помнить точные флаги.

**5 вещей, которые НЕЛЬЗЯ делать (все [C] Критический -- блокируют коммит):**

1. Нет эмодзи нигде (используйте текстовые теги `[OK]`, `[FAIL]`, `[TODO]`)
2. Нет Unicode иконок (используйте текстовые теги или SVG через `![]()`)
3. Нет типографских символов в заголовках, таблицах или коде (длинное тире `--`, копирайт, градус -- разрешены только в обычном тексте)
4. Нет псевдографики таблиц вне блоков кода (используйте Markdown таблицы `| ... |`)
5. Нет закрывающих `#` на заголовках, нет `*` или `+` для неупорядоченных списков (используйте только `-`)

**5 вещей, которые НУЖНО делать:**

1. Один H1 на документ, в начале
2. Каждый блок кода с ограждением указывает язык (используйте `text` или `bash`, если неизвестно)
3. Корневой README репозитория приложения заканчивается на `---\nBuilt with: <стек>`
4. Используйте маркер `(ref)` только когда символ является *объектом демонстрации* (например, в таблице "запрещенные символы")
5. Запускайте ESLint + `lint-md.js` перед коммитом (husky + lint-staged автоматизируют это)

## 1. Введение и Цели

Этот стандарт устанавливает правила форматирования Markdown документации для обеспечения визуальной согласованности и профессионального качества продукта.

**Цели:**

- Обеспечить визуальную согласованность документации
- Поддерживать профессиональное качество продукта
- Гарантировать контроль через систему дизайна
- Устранить неконтролируемые визуальные артефакты
- Автоматически применять стандарты через ESLint, где это возможно

## 3. Запрещенные Элементы

Запреты символов определены в **Политике без Unicode v2.3** (STD-DOC-003) разделы 4-5. Уровень **[C] Критический** применяется к файлам документации -- та же серьезность, что и для исходного кода (одно правило = одна серьезность).

**Сводка (авторитетные правила в STD-DOC-003):**

| Категория | Уровень | Примечание |
|-----------|---------|------------|
| Эмодзи | [C] | Без исключений |
| Unicode иконки | [C] | Используйте текстовые теги |
| Псевдографика таблиц | [W] | Используйте синтаксис Markdown |
| Типографика в заголовках/коде | [W] | Разрешена только в обычном тексте (см. область ниже) |

**Область запрета (для типографики в .md файлах):**

- Заголовки и подзаголовки
- Таблицы (кроме справочных таблиц - см. ниже)
- Inline код и блоки кода (комментарии, строки)
- Имена файлов и папок

**Исключение `(ref)` для справочных таблиц и блоков кода:** Если назначение ячейки таблицы или строки блока кода -- идентифицировать конкретный символ (показать, что запрещено или разрешено), символ может быть включен с маркером `(ref)`. Это не нарушает дух стандарта: символ используется как объект описания, а не как форматирование. Без реального символа пример теряет ясность: "Неправильно: `--` (ref) в заголовке" -- это демонстративно; "Неправильно: длинное тире в заголовке" -- это слепо.

**Где `(ref)` уместен:**

- В ячейке таблицы, назначение которой -- идентифицировать символ (например, колонка "Неправильно" §4.4 Текстовые Теги)
- В строке блока кода с ограждением, назначение которой -- продемонстрировать символ (например, `Неправильно: # Заголовок -- (ref) Подзаголовок` в §5.1)
- В абзаце документации, который явно обсуждает символ как объект (например, "Символ `--` (ref) это U+2014")

**Где `(ref)` НЕ уместен:**

- В обычном тексте, где символ используется как форматирование (например, "Установите зависимости -- затем запустите сборку" -- это просто длинное тире, маркер не нужен... но оно ВСЕ ЕЩЕ запрещено в заголовках/таблицах/коде, см. §3)
- В заголовке или ячейке таблицы, которая не посвящена самому символу
- Как способ проскользнуть с запрещенным символом в заголовок, добавив `(ref)` -- маркер не дает иммунитета, *назначение* ячейки должно быть идентификацией символа

## 5. Правила Форматирования

### 5.1. Заголовки

- Используйте `#` для H1, `##` для H2 и т.д.
- Не используйте закрывающие символы `#`
- Только один H1 на документ
- Не используйте типографские символы (такие как длинное тире) в заголовках

```text
Правильно:      # Заголовок
Неправильно:    # Заголовок #
Неправильно:    # Заголовок -- (ref) Подзаголовок
```

Символ `--` (ref) в примере выше -- это демонстрация запрещенного символа; маркер `(ref)` указывает на справочное использование.

### 5.2. Списки

**Неупорядоченные:**

- Всегда используйте `-` как единственный маркер для неупорядоченных списков.
- Не смешивайте с `*` или `+`.

```text
Правильно:      - Элемент 1
Неправильно:    * Элемент 1
Неправильно:    -> Элемент 1
```

**Упорядоченные:**

```text
1. Первый элемент
2. Второй элемент
```

### 5.3. Выделение Текста

| Формат | Синтаксис |
|--------|-----------|
| Полужирный | `**текст**` |
| Курсив | `*текст*` |
| Зачеркнутый | `~~текст~~` |

### 5.4. Форматирование Кода

**Inline код** (внутри текста):

```markdown
Используйте функцию `processFile()` для обработки.
```

**Блок кода** (с указанием языка):

````markdown
```typescript
const config = {
  encoding: 'utf-8',
  strict: true
};
```text
````

**Правило для Неизвестных Языков:**
Если точный язык программирования или формат не поддерживается рендерером или неизвестен, всегда указывайте `text` или `bash` вместо оставления блока пустым.

```text
Правильно:      ```text
Неправильно:    ```text
```

**Правила:**

| Элемент | Формат | Назначение |
|---------|--------|------------|
| Inline код | `` `код` `` | Функции, переменные, команды внутри текста |
| Блок кода | `` ```язык `` | Многострочный код, примеры, конфигурации |

**НЕ используйте пользовательские цвета:**

- Markdown не поддерживает цвета нативно
- HTML теги такие как `<span style="color:red">` могут быть заблокированы на GitHub
- Подсветка синтаксиса применяется автоматически рендерером
- Цвет -- ответственность темы (GitHub, VS Code), а не документа

### 6.1. Визуальные Элементы в Markdown

Правила иконок и графики определены в **разделе 7 STD-DOC-003**. Для .md файлов:

См. **раздел 7.4 STD-DOC-003** для требований к брендовым логотипам. При упоминании технологий в документации используйте официальные SVG через синтаксис изображений Markdown.

## 7. Бейджи

Бейджи -- это графические индикаторы метаданных проекта (версия, статус сборки, лицензия). Размещайте их в начале README.md после заголовка.

### 7.1. Разрешенные Источники

| Источник | Использование |
|----------|---------------|
| shields.io | Рекомендуется -- генерирует PNG/SVG без эмодзи |
| пользовательский SVG | Разрешен, если соответствует Политике без Unicode |

### 7.2. Запрещено

- Эмодзи в бейджах (shields.io поддерживает, но не используйте)
- Внешние иконки внутри бейджей
- Бейджи со статусами, не отражающими реальность

### 7.3. Типичные Бейджи для npm Пакета

```markdown
[![npm version](https://img.shields.io/npm/v/zai-agent-toolkit.svg)](https://www.npmjs.com/package/zai-agent-toolkit)
[![License: MIT](https://img.shields.io/badge/License-MIT-yellow.svg)](https://opensource.org/licenses/MIT)
```

### 7.4. Плейсхолдеры (для проектов без CI)

Если CI не настроен, используйте статические бейджи:

```markdown
[![Status: Draft](https://img.shields.io/badge/Status-Draft-yellow.svg)]()
[![Version: 1.9.5](https://img.shields.io/badge/Version-1.9.5-blue.svg)]()
```

Когда CI настроен, замените на динамические.

## 8. Сигнатура Стека

Файлы README **приложений** должны содержать сигнатуру стека в конце файла.

**Область (применимо к):**
- `README.md` в корне **репозитория приложения** (репозиторий, который поставляет запускаемый код -- например, приложение Next.js, библиотека, CLI инструмент)
- `CHANGELOG.md` в корне репозитория приложения (опционально, но рекомендуется)

**Область (НЕ применимо к):**
- **Стандарты** (`standards/standards/*.md`) -- они описывают правила, а не "построено на чем-то"
- **Правила** (`guard/rules/*.md`) -- то же
- **Скиллы** (`skills/skills/*/SKILL.md`) -- то же
- **Шаблоны** (`templates/*.md`) -- мета-документы, а не приложения
- **Оркестратор / мета-репозитории** (например, `Z-ai-governance`, `standards/`, `guard/`, `skills/`) -- они фиксируют директории и управляют, а не поставляют стек приложения
- **Вложенные документы** (`docs/**/*.md`) -- опционально, не требуется

Если файл неясен, тест: "Поставляет ли этот репозиторий запускаемое приложение, стек которого важен читателю?" Если да, корневой README должен иметь Сигнатуру Стека. Если нет (управление, стандарты, мета-репозитории), Сигнатура Стека запрещена -- это был бы культурный ритуал.

**Формат:**

```markdown
---
Built with: <технологии проекта>
```

Конкретный стек определяется проектом, а не стандартом. Пример для проектов Next.js:

```markdown
---
Built with: Next.js 16 + TypeScript + Tailwind CSS
```

Для значения по умолчанию в этом стеке см. `templates/README_TEMPLATE.md`.

**Правила:**

- Размещение: конец файла
- Разделитель: три дефиса `---`
- Содержание: буквы, цифры, знаки `+` и `:`
- Графика запрещена

## 9. Контроль и Применение

### 9.1. Уровень [C] Критический - Политика Применения

| Этап | Действие | Блокирует мерж? |
|------|----------|-----------------|
| Редактор | Inline ошибка в VS Code | н/a (в реальном времени) |
| Pre-commit | husky + lint-staged прерывает коммит | **Да** |
| CI Pipeline | Задача падает на `eslint --max-warnings=0` | **Да** |
| Преди мержом | Проверка статуса GitHub падает | **Да** |

**Правило:** Уровень [C] автоматически блокирует мерж. Чтобы внести PR с нарушением [C], автор должен:

1. Исправить нарушение (предпочтительно), ИЛИ
2. Добавить inline `eslint-disable-next-line <имя-правила>` с комментарием-обоснованием, И
3. Получить одобрение Tech Lead на PR

**Эскалация:** Отключение правил уровня [C] в продакшен-коде или документации без одобрения Tech Lead запрещено. Переопределение `lint-md.js --level=W` существует для разовой сортировки (например, аудит newly-imported codebase); оно не должно быть подключено к CI.

**Мягкий opt-out для унаследованных проектов:** Проект, который еще не готов к применению [C], может установить `lint-md.js --level=W` в своем локальном `package.json` скрипте во время окна миграции. Это решение является per-project, time-boxed и должно быть задокументировано в README проекта с целевой датой перехода. 默认 остается [C].

### 9.2. Обязательная Валидация Всех .md Файлов

**Правило:** Любой созданный или добавленный `.md` файл в проекте **должен** пройти валидацию и редактирование согласно этому стандарту.

| Действие | Когда | Ответственный |
|----------|-------|---------------|
| Создание нового .md | Перед коммитом | Автор |
| Добавление внешнего .md | Перед мержем | Ревьюер |
| Копирование .md из другого проекта | Перед коммитом | Автор |

**Процесс:**

1. Валидация по чеклисту (см. раздел 12)
2. Исправление нарушений
3. Добавление сигнатуры стека (если корневой файл)
4. Проверка в Code Review

Что изменилось:

1. Эмодзи удален из H1
2. Маркер списка `*` заменен на `-`
3. `[v]` / `[x]` заменены на `[OK]` / `[FAIL]`
4. Длинное тире в H2 заменено на двоеточие (`:`)
5. ASCII диаграмма обернута в fenced `text` блок кода (теперь это больше не "псевдографика таблиц в обычном Markdown" -- это теперь разрешенная ASCII диаграмма согласно §4.2 / §12)
6. Курсив внутри цитаты заменен на обычный текст (цитата + курсив не запрещены правилом, но хрупки в разных рендерерах; более безопасный паттерн показан здесь)

## 14A. Известные Проблемы и Предлагаемые Решения

Этот раздел документирует обнаруженные несоответствия, отсутствующее содержание и предлагаемые исправления. Каждая проблема имеет ID, статус и предлагаемое действие. Проблемы, решенные в текущей версии, помечены `[RESOLVED]`; нерешенные проблемы помечены `[OPEN]`.

### MD-001 `[RESOLVED в v2.4.0]` -- Противоречие уровней `[C]` и `[W]`

**Проблема:** Заголовок этого файла (§0) гласил `Level: **[C] Critical** (unified with STD-DOC-003 -- same rule = same severity)`. Подпись гласила `Document complies with MARKDOWN_STANDARD v2.3 (level [W])`. Реестр в `META-001-standard-id-system.md` (STD-META-001 §4.4) перечислял этот стандарт как `[W] Warning`. `ARCH-002-implementation-order.md` §1 таблица Group B также перечисляла его как `[W]`. Было трехстороннее противоречие: заголовок говорил `[C]`, подпись/реестр/IMPLEMENTATION_ORDER говорили `[W]`.

**Решение (v2.4.0):** Принят `[C] Critical` как авторитетный уровень. Обоснование: (1) принцип "одно правило = одна серьезность" из STD-DOC-003 §3 является замыслом дизайна v2.3.0; (2) конфигурация ESLint в §10 уже относилась к Markdown эмодзи/графике Unicode как `error` (не `warn`) согласно §10.6; (3) `lint-md.js` по умолчанию использует `--level=C`. Все четыре расположения теперь говорят `[C]`:
1. Заголовок этого файла (§0) -- без изменений, уже был `[C]`.
2. Подпись этого файла -- обновлена на `(level [C])`.
3. Запись реестра STD-META-001 §4.4 -- обновлена на `[C]`.
4. Таблица Group B §1 STD-ARCH-002 -- обновлена на `[C]`.

Кроме того, §9.1 был переписан с политики `[W] non-blocking` на политику `[C] blocking` с рабочим процессом `eslint-disable` + одобрение Tech Lead и оговоркой о мягком opt-out для унаследованных проектов. Описание §10.5.2, правило 3 §10.8, комментарий §10.9 и чеклист §13 были обновлены для согласованности.

### MD-002 `[RESOLVED в v2.3.1]` -- Внутри-документные ссылки на STD-DOC-003 цитировали v2.2

**Проблема:** До v2.3.1 поле `Related:` в заголовке гласило "No-Unicode Policy v2.2 (STD-DOC-003)". Ссылки внутри документа в §3, §4.1, §4.2, §6.1, §6.2, §6.3, §7.2, §7.4, §11.1 и §13 также цитировали v2.2. Фактическая версия STD-DOC-003 на момент аудита была v2.3.0.

**Решение:** Поле `Related:` в заголовке обновлено на "No-Unicode Policy v2.3 (STD-DOC-003)". Все внутри-документные ссылки на STD-DOC-003 v2.2 обновлены до v2.3.

### MD-003 `[RESOLVED в v2.4.0]` -- §10.3 Конфигурация ESLint устанавливает Markdown эмодзи на `warn`, противоречит таблице маппинга §10.6

**Проблема:** До v2.4.0 §10.3 (Конфигурация ESLint -- Flat Config) устанавливал `no-unicode-policy/no-emoji-in-md` на `"warn"` и `no-unicode-policy/no-unicode-graphics-in-md` на `"warn"` для файлов `**/*.md`. Однако §10.6 (таблица маппинга вложенных стандартов) перечислял те же правила с серьезностью `error` для STD-DOC-002 §3 (Запрещено: Эмодзи и Unicode иконки). Сноска таблицы §10.6 гласила "Правило, исходящее из STD-DOC-003 ([C] Critical), применяется с той же серьезностью независимо от контекста -- исходный код или документация." Это было внутренне непоследовательно: §10.3 использовал `warn`, §10.6 говорил `error`.

**Решение (v2.4.0):** Согласованы §10.3 и §10.4 с §10.6 путем изменения обоих `no-emoji-in-md` и `no-unicode-graphics-in-md` с `"warn"` на `"error"` как в flat config (§10.3), так и в legacy config (§10.4). Добавлена заметка в §10.3 с объяснением: "Серьезность: error ([C] Critical) -- одно правило, та же серьезность, что и исходный код. Чтобы смягчить применение для миграции унаследованного проекта, переопределите в локальном eslint.config.js проекта на `warn` с документированной датой перехода." Это согласовано с решением MD-001 (сохранить [C]).

### MD-004 `[OPEN]` -- `scripts/check-md.sh` обнаружил 5 ранее существовавших нарушений голых fence

**Проблема:** Новая обертка `scripts/check-md.sh`, добавленная в v2.4.1, запускает статические проверки по всему репозиторию при вызове без аргументов. Ее первый полный запуск по репозиторию выявил 5 голых code fences (блоки с ограждением без тега языка, нарушение §5.4):

- `README.md:14`
- `docs/verify-id-graph-spec-v1.0.md:32`
- `docs/verify-id-graph-spec-v1.0.md:60`
- `docs/verify-id-graph-spec-v1.0.md:337`
- `docs/verify-id-graph-spec-v1.0.md:452`

Эти нарушения предшествуют скрипту-обертке -- они не были введены v2.4.1, только обнаружены им. Они находятся в документации управления / спецификациях (не в корпусе стандартов), поэтому проскользнули мимо аудита дедупликации v2.3.0, который фокусировался на `standards/*.md`.

**Предлагаемое решение (отдельный PR):** Для каждого нарушения добавить соответствующий тег языка -- `bash` для шелл-фрагментов, `text` для обычных форматированных блоков или реальный язык (`javascript`, `yaml` и т.д.) где применимо. Исправление является механическим и не меняет семантику. До тех пор обертка сообщает об этих ошибках при полном запуске по репозиторию, но проходит при `bash scripts/check-md.sh standards/` (корпус стандартов сам по себе чист).
